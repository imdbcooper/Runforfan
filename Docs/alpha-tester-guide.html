1. Что тестируем

Runforfan в alpha должен быть проверен не только на отсутствие ошибок, но и на продуктовую понятность: пользователь должен понимать, почему появилась метрика, предупреждение, кандидат импорта или рекомендация по плану.

Импорт и валидация

Проверяем upload screenshots, CSV import, LLM/template recognition, candidate correction, confirm/reject и activity creation.

Профиль и зоны

Проверяем completeness, safety warnings, HR/pace/RPE zones, manual HR overrides и measurements.

Планы и календарь

Проверяем preview/generate, active plan, workout lifecycle, feedback, matching, reschedule и adherence.

Расчеты

Проверяем pace, VDOT, Riegel predictions, TRIMP, CTL/ATL/TSB, monotony, strain и execution score.

Safety

Проверяем conservative mode, injury/recovery flags, fatigue/load warnings и понятные ограничения продукта.

Данные

Проверяем export, audit log, LLM provider secrets и удаление данных через подтверждение DELETE.

2. Перед началом

Safety disclaimer

Runforfan не является медицинским устройством. Предупреждения в интерфейсе являются спортивными эвристиками. При боли, травме, необычной одышке, головокружении, обмороке или ухудшении самочувствия нужно прекратить тренировку и обратиться к специалисту.

Минимальная подготовка

Убедитесь, что backend отвечает на GET http://127.0.0.1:8080/health.
Откройте http://127.0.0.1:5173/admin/ в desktop browser.
Войдите через Telegram или dev/demo login.
Переключите язык ru/en и проверьте, что UI не ломается.
Подготовьте несколько скриншотов тренировок и один CSV-файл, если тестируете import pipeline.
Не используйте реальные production API keys в недоверенном alpha-окружении.

4. Сценарии тестирования

Must test

A. Первый вход и профиль

  1. Войдите в admin UI.
  2. Откройте Profile & zones.
  3. Заполните дату рождения или HRmax, resting HR, threshold pace или threshold HR, вес, preferred weekdays и max duration.
  4. Добавьте injury/health/recovery данные, если тестируете safety warnings.
  5. Сохраните профиль и проверьте zones.

Ожидается: completeness растет, zones recalculated, safety warnings понятны, onboarding исчезает при достаточной готовности.

Must test

B. Импорт скриншотов

  1. Откройте Imports.
  2. Загрузите 1-6 изображений PNG/JPEG/WEBP.
  3. Проверьте recognition message и batch status.
  4. Если появился candidate, исправьте поля и нажмите confirm.
  5. Откройте созданную activity в Activities.

Ожидается: batch виден в history, candidate требует подтверждения при uncertainty, created activity имеет validation и source metadata.

Core

C. CSV import

  1. Откройте Imports.
  2. Выберите CSV с title, started_at, duration, distance или HR.
  3. Проверьте counters created, duplicates, matched, failed.
  4. Повторно загрузите тот же CSV для проверки duplicate detection.

Ожидается: дубликаты не создают новые активности, ошибки строк понятны, created rows появляются в Activities.

Core

D. Manual activity

  1. Откройте Activities.
  2. Заполните title, started, type, duration, distance и avg HR.
  3. Создайте activity.
  4. Откройте detail и измените duration/distance через quick edit.

Ожидается: duration required, pace/speed пересчитываются backend-ом, derived metrics/load обновляются.

Must test

E. Plan preview и создание плана

  1. Откройте Plans.
  2. Заполните goal, distance, target date, available days, current weekly volume, support sessions и constraints.
  3. Нажмите Preview.
  4. Проверьте baseline, risk flags, volume curve и first workouts.
  5. Создайте draft или active plan.

Ожидается: Preview не сохраняет план, Generate создает план, Dashboard/Calendar видят active plan.

Core

F. Workout lifecycle

  1. В плане откройте тренировку.
  2. Проверьте blocks, target, purpose и safety note.
  3. Заполните manual completion или найдите activity candidate.
  4. Сохраните feedback: RPE, soreness, pain, sleep, notes.

Ожидается: workout получает status done, появляется execution score, adherence обновляется.

Core

G. Calendar matching

  1. Откройте Calendar.
  2. Проверьте This week, Next week, This month и custom range.
  3. Перенесите workout, отметьте missed/skipped и восстановите planned.
  4. Найдите activity/workout candidates и выполните link.

Ожидается: range больше 42 дней отклоняется, hard-session warnings видны, matching score/reasons понятны.

Must test

H. Analytics, load и zones

  1. Откройте Analytics и переключите периоды.
  2. Проверьте weighted pace, avg HR, adherence, best efforts и insights.
  3. Откройте Load & Recovery и проверьте CTL/ATL/TSB.
  4. Откройте Zones Analytics и проверьте coverage, Seiler split, planned vs actual.

Ожидается: Why? объясняет расчет, custom range работает, empty states не выглядят как ошибка.

Core

I. Performance и race goal

  1. Откройте Performance.
  2. Добавьте race или time trial не короче 3 км.
  3. Проверьте VDOT source, predictions, PBs, threshold trend и pace zones.
  4. Создайте race goal в Goals & races.

Ожидается: noisy source помечается, predictions имеют confidence/warnings, race goal показывает readiness и prediction range.

Core

J. Settings, LLM и export

  1. Откройте Settings & data.
  2. Добавьте OpenAI-compatible или Anthropic provider.
  3. Проверьте Test provider и Set default.
  4. Скачайте JSON export и activities CSV.
  5. Проверьте audit log.

Ожидается: API key не возвращается в UI/export, provider test сообщает status/message, exports не содержат secrets.

5. Формулы и расчеты, которые нужно понимать

Формулы ниже нужны не для ручной перепроверки каждого значения, а для корректной оценки поведения продукта, объяснений Why?, confidence и safety warnings.

Pace, speed, weighted pace

pace = duration_seconds / distance_km
speed_kmh = distance_km / (duration_seconds / 3600)
weighted_pace = sum(duration) / sum(distance)

Weighted pace используется в analytics и segment summaries, чтобы короткие тренировки не искажали средний темп.

Estimated HRmax, Tanaka

HRmax = round(208 - 0.7 * age)

Fallback, если нет измеренного HRmax. Не равен лабораторному измерению и должен иметь lower confidence.

HRR zones, Karvonen

HRR = max_hr - resting_hr
zone_hr = resting_hr + percent * HRR

Требует resting HR и max HR. Используется для персонализированных HR zones.

Pace zones

easy      = threshold +45..+95 sec/km
steady    = threshold +20..+44 sec/km
threshold = threshold -5..+10 sec/km
interval  = threshold -35..-6 sec/km
rep       = threshold -60..-36 sec/km

Threshold pace берется из профиля или fallback из performance source.

VDOT

velocity = distance_m / time_min
vo2 = -4.60 + 0.182258*v + 0.000104*v^2
percent = 0.8 + 0.1894393*e^(-0.012778*t)
        + 0.2989558*e^(-0.1932605*t)
VDOT = vo2 / percent

Используется только для race-like или time trial sources. Easy run не должна становиться главным VDOT source.

Riegel prediction

predicted_time = source_time *
  (target_distance / source_distance) ^ 1.06

Экстраполяция за пределы отношения 0.25-4 снижает confidence и должна давать warning.

TRIMP

hr_ratio = (avg_hr - resting_hr) / (max_hr - resting_hr)
male:   load = min * ratio * 0.64 * exp(1.92 * ratio)
female: load = min * ratio * 0.86 * exp(1.67 * ratio)

Требует sex male/female и корректные HR inputs. Иначе используется другой load method.

sRPE load

sRPE = duration_minutes * RPE_0_10

Используется, если workout feedback содержит RPE. Это субъективная оценка нагрузки.

CTL, ATL, TSB

alpha = 1 - exp(-1 / tau_days)
EWMA = previous + alpha * (load_today - previous)
CTL tau = 42 days
ATL tau = 7 days
TSB = CTL - ATL

Это эвристики fitness/fatigue/form. При истории меньше 42 дней должен быть warmup warning.

Monotony and strain

monotony = mean(daily_loads) / stddev(daily_loads)
strain = sum(daily_loads) * monotony

Высокая monotony означает слишком однообразную нагрузку. При нулевой вариативности расчет может быть недоступен.

Adherence

completion_rate = done_workouts / planned_workouts

Используется в Dashboard, Plans, Goals. Missed/skipped тренировки снижают completion rate.

Execution score

volume_score = 1 - abs(actual/planned - 1) / 0.5
final_score = average(volume, intensity, subjective)

Pain, high fatigue, high RPE и плохой сон ограничивают subjective component.

9. Known limitations

Alpha status

Функции, UX и формулы могут меняться. Нужны баг-репорты по продуктовой понятности, не только по crashes.

LLM recognition

Качество зависит от provider, модели, screenshots и prompt. LLM может ошибаться, поэтому confirmation обязателен.

Training load

CTL/ATL/TSB, monotony и strain являются эвристиками. Они не предсказывают здоровье или травмы.

Performance

VDOT и Riegel predictions требуют качественного race/time trial source. Noisy, старые и короткие results снижают confidence.

Zones

Зоны зависят от профиля. Без HRmax/resting HR/threshold pace часть расчетов будет недоступна.

Matching

Auto-match отклоняет неоднозначные candidates. Это ожидаемо, пользователь должен выбрать link вручную.
